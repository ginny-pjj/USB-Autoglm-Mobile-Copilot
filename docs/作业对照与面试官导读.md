# 作业对照与面试官导读（USB v1）

## 一、与 Open-AutoGLM 官方作业的对照

本项目基于官方仓库：[zai-org/Open-AutoGLM](https://github.com/zai-org/Open-AutoGLM)

| 官方作业要求 | USB v1 如何实现 | 对应位置 |
| --- | --- | --- |
| Python 3.10+ 环境 | Windows 本地运行 FastAPI + Open-AutoGLM | `server/`、`Open-AutoGLM/` |
| ADB 连接 Android 7.0+ 设备 | USB 数据线 + `adb devices` | `server/connect_phone.bat` |
| 开启 USB 调试、安装 ADB Keyboard | 文档说明 + 提供 `ADBKeyboard.apk` | 根目录、`docs/faq.md` |
| 调用智谱 BigModel（autoglm-phone） | `.env` 配置 `BIGMODEL_API_KEY` | `server/.env.example` |
| 自然语言控制手机完成任务 | App 输入任务 → PhoneAgent 循环执行 | `mobile-app/`、`Open-AutoGLM/main.py` |
| phone_agent 内核（截图→VLM→ADB） | 保留官方结构，未重写 Agent 核心 | `Open-AutoGLM/phone_agent/` |
| 命令行执行任务 | 后端 subprocess 调用 `main.py` | `server/main.py` |
| Observe / Think / Act 闭环 | `agent.py` 主循环 + `handler.py` 动作执行 | 见 `docs/architecture.md` |

### 本项目的工程化扩展（超出官方 CLI 作业）

| 扩展点 | 价值 |
| --- | --- |
| FastAPI 任务服务 | 把 CLI 封装成 HTTP API，便于 App 调用 |
| Android 控制端 App | 移动端发任务、看进度、看 Trace |
| Mock / Real 双模式 | Mock 联调 UI，Real 跑真实 Agent |
| 结构化 Trace | 将原始日志清洗为 OBSERVE / THINK / ACTION / RESULT |
| `adb reverse` 本地访问 | 手机 App 填 `127.0.0.1:8000` 即可连电脑后端 |

## 二、30 秒项目介绍（面试可直接背）

> 我基于 Open-AutoGLM 做了 AutoGLM Mobile Copilot。官方是命令行 Phone Agent，我加了 FastAPI 后端和 Android App，形成「App 发任务 → 后端调度 → 截图 + 视觉大模型决策 → ADB 执行 → Trace 可视化」的闭环。USB v1 是最稳定的第一版：电脑跑后端，USB 连手机，用 `adb reverse` 让 App 访问本地服务。我没有重写 Agent 内核，主要做工程封装和问题排查。

## 三、面试官常问 5 题

**Q1：你和官方 Open-AutoGLM 有什么区别？**

> 官方提供 CLI 和 `phone_agent` 内核；我加了 `server/`（FastAPI）和 `mobile-app/`（Android App），把 Agent 产品化，并做了 Trace 展示和 Mock/Real 模式。

**Q2：Agent 一步怎么执行的？**

> `agent.py` 每轮：ADB 截图 → 智谱 VLM 看图决策 → `handler.py` 解析动作（Tap/Launch/Type 等）→ ADB 执行 → 再截图，直到 finish 或超步数。

**Q3：为什么用 `adb reverse`？**

> 手机 App 访问 `127.0.0.1` 时，ADB 会把请求转发到电脑的 8000 端口，这样不用查电脑局域网 IP，USB 场景最省事。

**Q4：运行时电脑要一直开着吗？**

> USB v1 要。后端在本地 Windows 跑，`start_server.bat` 不能关。云端版可以把后端迁到服务器。

**Q5：你实际跑通了什么？**

> App 能发任务、Real 模式能连 USB 设备、设置/浏览器类任务能跑通；也排查过 API Key、ADB 授权、中文输入等问题。
